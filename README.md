Веб-приложение на React и TypeScript: пользователь вводит задачу, а несколько "агентов" по очереди планируют, выполняют подзадачи и собирают итог. Логика симулируется (задержки, заготовленные ответы), зато видно цепочку взаимодействия и ленту сообщений.

Запуск

npm install
npm run dev

Откройте в браузере http://localhost:5173

Архитектура (кратко)

- src/App.tsx — состояние запуска (RunSnapshot), кнопки старт/стоп/очистка, при каждом событии симуляции обновляет UI
- src/agents/ — ядро симуляции:
  - types.ts — типы задач/сообщений/событий + контракты AgentImpl/AgentAction (event-driven модель)
  - presets.ts — описание агентов
  - engine.ts — доставляет сообщения в inbox агентов, применяет actions, ведёт snapshot; остановка через AbortController
  - impl/ — логика агентов:
    - planner.ts — декомпозиция задачи, запрос bid, выбор исполнителя (decision) с объяснением
    - worker.ts — ставки и выполнение подзадач
    - builder.ts — ставки, выполнение, сборка черновика
    - reviewer.ts — ревью черновика и финализация
    - utils.ts — эвристики (keywords/регулярки) для разбиения текста на подзадачи
- src/components/dashboard/ — UI карточек задач и строк ленты
- src/components/flow/ — граф взаимодействия агентов
- src/lib/ — утилиты

Как устроена система агентов

Про ключевые слова:

Декомпозиция пользовательского текста делается простой эвристикой: в src/agents/impl/utils.ts есть набор keywords (регулярные выражения) для тем вроде ui, архитектуры, визуализации, запуска и тестов. Planner использует эти совпадения, чтобы предсказуемо (и объяснимо) сформировать список подзадач.

1. Planner (планировщик) читает текст задачи от пользователя, разбивает его на подзадачи (по ключевым словам или по предложениям) и назначает каждую подзадачу исполнителям
2. Researcher и Builder по очереди "берут" свои подзадачи: в интерфейсе это выглядит как смена статусов и сообщения вроде "взял в работу" / "готово". Содержимое ответов заранее шаблонное — видимая цепочка
3. Builder дополнительно склеивает результаты подзадач в черновик цельного ответа
4. Reviewer смотрит на черновик, добавляет замечания (тоже из заготовки) и выдаёт финальный текст, который показывается отдельным блоком в UI

Между шагами стоят случайные паузы
